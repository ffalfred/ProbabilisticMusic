# V2 — Expressive Interpretation Engine

> **Beta.** V2 is functional but the musical priors in `v2/transition_table.yaml` are starting estimates — you may want to tune them over time.

---

## What is V2?

In standard mode (V1), the tool renders your score the same way every time (unless you use probabilistic parameters). Every event has the same loudness, timing, and effects each run.

V2 adds a layer of **musical interpretation** on top. It reads the `dynamics:` markings in your score — marks like `p`, `f`, `sfz`, `cresc` — and decides *how to interpret them expressively* before rendering. The result is different every run: different phrasing, slightly different timing, different amounts of reverb, different attack shapes.

The variation is not random noise. It's driven by a model of musical performance: how real performers react to dynamic context, phrase position, and the history of what they just played.

---

## How to turn V2 on

### In the editor

At the bottom of the editor, find the **Engine** dropdown and select **V2 β**.

### From the terminal

Open `config.yaml` (in the ProbabilisticMusic root folder) in any text editor and change:

```yaml
engine: v2
```

## Adding dynamics to your score

V2 needs a `dynamics:` section in your score. Without it, V2 uses a neutral default for every event.

```yaml
dynamics:
  - t: 0.0
    marking: p

  - t: 10.0
    marking: mp

  - t: 19.0
    marking: f

  - t: 21.0
    marking: sfz

  - t: 24.0
    marking: subito_p

  - t: 30.0
    marking: cresc
    duration: 8.0        # how long the crescendo lasts in seconds

  - t: 38.0
    marking: ff
```

> Note: V2 uses `marking:` (not `mark:` like in V1 dynamics). Both can coexist in the same score.

## Supported dynamic markings

### Volume levels (soft to loud)

| Marking | Meaning |
|---------|---------|
| `ppp` | very very quiet |
| `pp` | very quiet |
| `p` | quiet |
| `mp` | medium quiet |
| `mf` | medium loud |
| `f` | loud |
| `ff` | very loud |
| `fff` | very very loud |

### Expressive markings (instantaneous)

| Marking | What V2 does with it |
|---------|----------------------|
| `sfz` | Sharp accent. Then the next 1–2 events are pulled quieter — as if the performer is withdrawing after the outburst. |
| `fp` | Loud attack that immediately collapses to soft. |
| `subito_p` | Sudden jump to quiet with no transition. |
| `subito_f` | Sudden jump to loud with no transition. |

### Gradual (require `duration:`)

| Marking | What it does |
|---------|-------------|
| `cresc` or `crescendo` | Gradually gets louder over `duration` seconds |
| `decresc`, `decrescendo`, or `diminuendo` | Gradually gets quieter over `duration` seconds |

## What V2 changes per event

For each event, V2 samples a set of small adjustments and applies them before V1 renders:

| Parameter | Applied as |
|-----------|-----------|
| `gain_db` | Small loudness offset added to the event |
| `timing_offset_ms` | Slight timing shift forward or backward (humanises the rhythm) |
| `reverb_wet` | Adds or adjusts reverb on the event |
| `attack_shape` | Makes the fade-in longer or shorter |
| `brightness` | Small EQ boost or cut at 5kHz |

The size of these adjustments depends on the dynamic context and the model's history.

---

## Markov modes

V2 uses a statistical model called a Markov chain to decide how to interpret each event based on what came before.

### Symbolic mode

The model looks at the score's markings and draws a random performance from them. Two runs see the same dynamic sequence but come out differently — the variation is independent each time.

### Joint mode (default, recommended)

The model also remembers what it actually rendered in previous events. An early loud draw pushes subsequent events toward loudness. A run that starts quiet tends to stay quieter. Each run develops its own performance character. More organic, more interesting over long pieces.

## config.yaml reference

The file `config.yaml` in the ProbabilisticMusic root folder controls V2's behaviour. Open it with any text editor:

```yaml
# v1 = standard rendering, v2 = expressive interpretation
engine: v2

# symbolic: only looks at score markings
# joint:    remembers what it rendered (more organic)
markov_mode: joint

# how many past events to consider (2 is a good default)
markov_order: 2

# diagonal = parameters vary independently
# full = parameters can correlate with each other
covariance: diagonal

# reset: history resets at each phrase boundary (recommended)
# continuous: history carries across the whole piece
phrase_boundary: reset

# how strongly old history influences current choices
# 0.7 is a good default
history_decay: 0.7

# what V2 assumes before it has any history
cold_start_marking: mf
cold_start_outputs:
  gain_db: -12.0
  brightness: 0.5
  timing_offset_ms: 0.0
  attack_shape: 0.5
  reverb_wet: 0.3

# set an integer to get the exact same result every time
# null means random
seed: null

# Bayesian update with MAESTRO data — not implemented yet
bayesian_update: false
maestro_path: null
```

## Reproducible runs

To get the same result every time:

```yaml
seed: 42     # any integer works
```

To get a different result every time:

```yaml
seed: null
```

When using the editor, you can also set the seed in the V2 sub-panel without touching `config.yaml`.

## Phrase boundaries

V2 automatically detects where phrases end (by looking for gaps between events). At each phrase boundary, the Markov history can either reset (recommended: each phrase starts fresh) or carry over (the previous phrase influences the next).

This is set by `phrase_boundary: reset` or `phrase_boundary: continuous` in `config.yaml`.

---

## Expected musical behaviours

**sfz recovery shadow** — after an `sfz` marking, the next 1–2 events are slightly quieter and duller than they would otherwise be. This fades: the first event after sfz gets the full shadow, the second gets half.

**cresc/decresc** — events inside a crescendo or decrescendo span have their gain shifted at a rate sampled from the model. The rate varies between runs — a restrained run may not crescendo as much; an expressive run may overshoot.

**Joint mode self-reinforcement** — a run that happens to draw loud values early will tend to stay louder. Quiet runs stay quiet. This gives each performance a consistent personality rather than random variation event by event.

## Tuning the transition table

`v2/transition_table.yaml` contains the starting priors for every dynamic transition — how loud, how bright, how much reverb, etc.

- **Larger `std`** → more variation between runs, more expressive
- **Smaller `std`** → tighter, more controlled, more consistent

To tune: render the same score 5–10 times with `seed: null` and listen. If it all sounds the same, increase `std` values. If it sounds chaotic, reduce them. The most impactful knob is `gain_db.std`.
