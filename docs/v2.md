# V2 — Expressive Interpretation Engine

> **Beta.** V2 is functional but the expert priors in `v2/transition_table.yaml` are starting estimates — expect to tune them as you work.

V1 renders a score deterministically (modulo probabilistic parameters). V2 adds a second layer of variation: it interprets the score's dynamic markings and decides *how expressive* each event should be before V1 renders it.

Every run of the same score produces a different but musically coherent result. The variation comes from the model's own performance history — not just random noise.

---

## What V2 does

For each event, V2 samples an **output vector**:

```
o(t) = [gain_db, brightness, timing_offset_ms, attack_shape, reverb_wet]
```

These values are drawn from a multivariate Gaussian whose mean and variance depend on the dynamic context. They are then applied to the event before V1 renders it:

| Parameter | Applied as |
|-----------|-----------|
| `gain_db` | Added to the event's `gain_db` |
| `timing_offset_ms` | Shifts the event's start time (humanises the grid) |
| `reverb_wet` | Injects or scales a reverb FX on the event |
| `attack_shape` | Scales the fade-in duration (longer = softer attack) |
| `brightness` | Injects a parametric EQ boost/cut at 5kHz |

## Dynamic markings

Add a `dynamics:` block to your score YAML. Each entry has a time and a marking:

```yaml
dynamics:
  - t: 0.0
    marking: p

  - t: 10.0
    marking: mp

  - t: 19.0
    marking: f

  - t: 21.0
    marking: sfz

  - t: 24.0
    marking: subito_p

  - t: 30.0
    marking: cresc
    duration: 8.0        # grows over 8 seconds

  - t: 38.0
    marking: ff
```

### Supported markings

#### Dynamic ladder (soft → loud)
| Marking | Meaning |
|---------|---------|
| `ppp` | pianississimo |
| `pp`  | pianissimo |
| `p`   | piano |
| `mp`  | mezzo-piano |
| `mf`  | mezzo-forte |
| `f`   | forte |
| `ff`  | fortissimo |
| `fff` | fortississimo |

#### Expressive (instantaneous)
| Marking | Behaviour |
|---------|-----------|
| `sfz` | Sforzando — sharp accent, then a quiet recovery shadow over the next 1–2 events |
| `fp` | Forte-piano — loud attack, immediately collapses to soft |
| `subito_p` | Sudden jump to soft with no transition |
| `subito_f` | Sudden jump to loud with no transition |

#### Gradual (require `duration:`)
| Marking | Behaviour |
|---------|-----------|
| `cresc` / `crescendo` | Gradual gain increase over `duration` seconds |
| `decresc` / `decrescendo` / `diminuendo` | Gradual gain decrease over `duration` seconds |

## Markov modes

V2 uses a **higher-order Markov chain** to decide what o(t) should be, conditioned on recent history.

### Symbolic mode

The chain only sees the score's markings. Each run varies because o(t) is sampled from a Gaussian — but the context is the same every run. Two runs see the same dynamic sequence but draw different values. Variation is **independent between runs**.

### Joint mode (default)

The chain also sees what was actually rendered in previous steps. An early loud draw shifts subsequent emissions. The piece develops its own performance personality during each run. Variation is **cumulative and self-reinforcing** — more organic, more unpredictable in long pieces.

---

## Phrase boundaries

V2 automatically infers phrase boundaries from the event structure (gaps between events larger than 3× the median inter-event gap). At each phrase boundary, behaviour depends on the `phrase_boundary` setting:

- **`reset`** — Markov history is wiped; each phrase starts fresh
- **`continuous`** — history accumulates across the whole piece; a loud phrase casts a shadow into the next

## config.yaml reference

Located at `beta_interpreter/config.yaml`. Used by both the CLI and the editor.

```yaml
# Engine: v1 (deterministic) or v2 (expressive interpretation)
engine: v2

# Markov mode: symbolic or joint
markov_mode: joint

# How many previous steps to condition on (1–4, default 2)
markov_order: 2

# Covariance: diagonal (parameters vary independently)
#             full (parameters co-vary — requires correlations in transition_table.yaml)
covariance: diagonal

# Phrase boundary behaviour: reset or continuous
phrase_boundary: reset

# History decay (joint mode): 1.0 = all steps equally, 0.5 = older steps less influential
history_decay: 0.7

# Cold start: what V2 assumes before any marking has been seen
cold_start_marking: mf
cold_start_outputs:
  gain_db: -12.0
  brightness: 0.5
  timing_offset_ms: 0.0
  attack_shape: 0.5
  reverb_wet: 0.3

# Set to an integer for reproducible runs; null = random
seed: null

# Stochastic V1 parameters (applied even in V1 mode)
v1_stochastic:
  speed:   {enabled: true,  std: 0.05}
  gain_db: {enabled: true,  std: 1.0}
  timing:  {enabled: false, std: 0.0}

# Bayesian update with MAESTRO data — not yet implemented
bayesian_update: false
maestro_path: null
```

## Running V2 from the CLI

1. Set `engine: v2` in `beta_interpreter/config.yaml`.
2. Add a `dynamics:` block to your score YAML.
3. Run as usual:

```bash
cd beta_interpreter
python main.py -i /path/to/audio.wav -s scores/my_score.yaml
```

Output files are named automatically and numbered per run:

```
output/output_<score_name>_<base_name>_001.wav
output/output_<score_name>_<base_name>_002.wav
...
```

Set `seed: 42` in `config.yaml` to reproduce a specific run.

---

## Using V2 in the editor

1. Launch the editor as usual (`python server.py`).
2. In the bottom bar, open the **Engine** dropdown and select **V2 β**.
3. The V2 sub-panel appears:
   - **mode** — `joint` (default) or `symbolic`
   - **seed** — leave blank for random; enter an integer for reproducible renders
   - **order** — how many past steps to consider (default 2)
4. Press **▶ Mix** to render. Each press produces a different result (unless seed is fixed).
5. When you **Export YAML**, the V2 config is saved inside the file under `_v2_config:`.

## Musical behaviours to expect

### Sfz recovery shadow
After an `sfz` marking, the next 1–2 events are pulled quieter and duller than they would otherwise be — as if the performer is withdrawing after an outburst. The shadow fades linearly: full on the next event, half on the one after.

### Cresc/decresc interpolation
For events inside a crescendo or decrescendo span, `gain_db` is adjusted linearly at a rate sampled from the transition table (roughly ±1.5 dB/sec by default). The rate itself varies between runs — a restrained performer may crescendo less than the score implies; an expressive one may overshoot.

### Phrase boundary reset
With `phrase_boundary: reset`, V2 reinitialises its history at each detected phrase boundary. The first few events of each phrase sound as if following a neutral `mf` passage regardless of how the previous phrase ended. This keeps phrases structurally independent.

### Joint mode self-reinforcement
With `markov_mode: joint`, a run that happens to draw loud values early will tend to sustain that energy — subsequent emissions are nudged in the same direction by the history blend. Quiet runs tend to stay quiet. This creates a more coherent performance character per run rather than random fluctuation.

## Tuning the transition table

`beta_interpreter/v2/transition_table.yaml` contains all the expert priors. Each transition defines `mean` and `std` per output parameter:

- **Larger `std`** → more expressive freedom, wider variation between runs
- **Smaller `std`** → tighter, more controlled, more consistent

Start by rendering the same score 5–10 times with `seed: null` and listening. If variation is too subtle, increase `std` values. If it sounds chaotic, reduce them. The most musically significant knobs are `gain_db.std` (loudness variation) and `timing_offset_ms.std` (rhythmic looseness).
